Graças ao aprimoramento das implementações, dos rascunhos desenvolvidos pelo consórcio W3C e diversas outras entidades e toda a comunidade de desenvolvedores que contribuem ativamente para o aperfeiçoamento tecnológico e inovativo da web, hoje é possível atingir níveis decentemente seguros de criptografia, para casos de uso específicos, mesmo na ponta do cliente (navegador), sem nem mesmo exigir processamento do lado do servidor. Isso é especialmente útil para situações em que a validação do acesso não é uma necessidade, em que os conteúdos não serão amplamente divulgados. Mais detalhes da implementação e bibliotecas utilizadas serão os tópicos seguintes deste artigo.

Vale constar que, a fim de evitar brecha, e também pela própria natureza do método, NÃO é possível recuperar posteriormente a senha definida para a proteção dos conteúdos. Também é totalmente não recomendado que se guarde ou armazene a senha em formato de texto cru em qualquer meio, digital ou físico. Carteiras de palavras-chave criptografadas, como a KWallet ou mesmo a KeePass, podem servir para o propósito de praticidade, mantendo algum nível de segurança.

Criptografia de senhas: Argon2

Conforme é possível perceber, o elo mais fraco, o qual requer maior interação humana para o método (estando sujeito a falhas), é a senha. Desse modo, diz-se que, embora métodos ou algoritmos de encriptação de dados sejam bastante eficientes e, alguns, até aqui jamais vazados (como o AES-256), métodos ainda mais avançados podem e devem ser usados para o armazenamento de senhas ou palavras-chave. Tendo isso em vista, esse projeto e outros os quais exigirem esse requisito, utiliza(rá) o algoritmo Argon2 para a criptografia de senhas. Esse algoritmo foi vencedor da Password Hashing Competition, uma competição de nível global que ocorreu em diversos anos, desde 2013 até 2015, na busca por um método de hashing (criptografia de via única) plenamente seguro para o armazenamento de senhas, devido a vazamentos ocorridos na década. Até hoje, o algoritmo é considerado forte e amplamente recomendado por comunidades de desenvolvimento de todo o mundo. A sua implementação aqui utilizada, para a linguagem JavaScript, executável em navegadores da web, foi desenvolvida e é mantida por @antelle (GitHub).

Criptografia de dados: AES-256

No entanto, para manter o conteúdo dos artigos plenamente seguro, não basta que se mantenha a senha ilegível, totalmente indecifrável, mas também que se vincule, de alguma forma, essa senha a um conteúdo igualmente criptografado: seus dados também precisam ser ilegíveis. Desse modo, a palavra-chave, validada pelo algoritmo Argon2, no próprio computador do cliente, serve para decodificar conteúdo criptografado utilizando a notória e amplamente conhecida especificação de algoritmos AES, desenvolvida pela agência estadunidense NIST em 2001. Em sua variação de tamanho de chave de 256 bits, o modelo é considerado totalmente seguro, e é utilizado até mesmo por agências de inteligência oficiais como a estadunidense NSA, e parte do corpo técnico de normas ISO (International Organization for Standardization). Desse modo, garante-se que o conteúdo de artigos não pode ser lido sem o preenchimento da senha correta. A implementação utilizada é de um pacote amplamente conhecido e popular entre projetos de aplicações web chamado CryptoJS, ativamente mantido.

Se você tem alguma dúvida da força de chaves de 256 bits de tamanho, considero que assista ao vídeo abaixo (legendas em português disponíveis!). Vale constar: o algoritmo Argon2 também salva suas senhas com um tamanho de chave de 256 bits, pelo menos em projetos nos quais eu for seu desenvolvedor. Ou seja, apenas escolha uma senha forte e estará, certamente, seguro!



Thanks to the enhancement of implementations, of the drawns developed by W3C consortium and a lot of other entities as well as all the developers community that contribute actively for the technological enhancement and innovation on web, now we are able to achieve decent levels of safety on cryptography, for some specific use-cases, even only in the client-side (browser), without any server processing. This is specially useful when access validation and register are not a need, when contents are not broadly shared. More details about implementation and utilized libraries will be the next topics of this article.

It is good to notice that, in order to avoid any leak, and for the method's nature itself, it is NOT possible to recover any password used for content's protection afterwards. Also it is totally unrecommended to keep or store the password in plain text format under any mean, digital or physical. Key wallet managers, like KWallet or even KeePass, could be used for practicality purpose, while keeping some security level.

Password cryptography: Argon2

As it is possible to notice, the weakest link, that requires the most human interaction for this method (being subject to fails), is the password itself. Said that, despite some algorithms and methods for data encryption being very efficients, with many that were never leaked until now (like AES-256), methods that are even more advanced can and must be used for password storing and validation. Looking at this, this project and others that will ask for this requirement, will utilize the Argon2 algorithm for password cryptography. This algo was the winner for Password Hashing Competition, a global level competition that happened for more than a year, between 2013 and 2015, searching for a viable hashing (one-way cryptography) method for password storage, answering some demands after password leakage episodes. The implementation used in this, for JavaScript language, executable in web browsers, was developed and is maintained by @antelle (GitHub).

Data cryptography: AES-256

However, to keep the article's contents fully safe, it is not enough to keep the password unreadable, but to link, by some mean, this password to a content equally unreadable. This way, the password, validated by Argon2 algorithm, in the client's computer itself, serves the purpose of decode the encrypted content utilizing the notable and broadly known AES specification, developed by the US agency NIST in 2001. In its 256-bits variation of key size, the model is considered totally secure, being utilized even by some official intelligence agencies like the NSA, as well as being part of the ISO (Internacional Organization for Standardization) technical standards corpus. This way, it is guaranteed that article's contents can't be read without inputting the right password. The used implementation is of a well-known package for web applications called CryptoJS, actively maintained until now.
